There's a rite of passage in graphics programming: drawing your first triangle.

**This usually takes thousands of lines of code.** Pick any tutorial for OpenGL, Vulkan, or raw WebGPU and you'll spend an afternoon negotiating pixel formats, allocating command pools, and wiring descriptor sets before a single colored pixel reaches the screen.

We're going to do it in **six** lines. Then we'll grow that one triangle into a fifteen-hundred-particle scene with a filmic post-processing stack — and along the way meet every object in the FragmentColor core API, plus the [shader catalog](/shaders/) we'll keep pulling from.

This tutorial walks through five steps. Each one adds one new tool. By the end, you'll have a fully composed multipass scene running in a single render call, and you'll know how to grow it.

## Step 1 — Three lines that draw

To draw anything anywhere, we need two things: a **surface** to draw to, and **something that draws**. In FragmentColor those are a `Target` (wraps a canvas or window) and a `Renderer`.

```rust
// canvas is platform-specific
let renderer = Renderer::new();
let target = renderer.create_target(canvas).await?;
```

Surface ready. Now the something-that-draws. A `Shader` is a tiny program that runs once per pixel — the smallest unit FragmentColor can render. Pick one off the public registry:

```rust
let shader = Shader::new("https://fragmentcolor.org/triangle.wgsl")?;
shader.set("color", [0.95, 0.30, 0.42, 1.0])?;
renderer.render(&shader, &target);
```

Congratulations. You just drew your first triangle.

(Or maybe not your first. You probably remember that one fondly.)

These three lines look unremarkable. Behind them, FragmentColor validated the WGSL, compiled a shader program, built a render pipeline, allocated GPU buffers, created bind groups, set up a render pass, and dispatched the draw call. Our shader runs **once for every pixel, every frame**. On a 1080p surface at 60 fps, that's roughly 125 million invocations per second, all in parallel.

<Demo file="01-hello-triangle/01-three-lines/example.rs" />

The shader at that URL declares one `color` uniform and bakes its three vertex positions into the program. FragmentColor parsed the WGSL, saw the uniform, and made it settable by name — that's how `shader.set("color", ...)` works.

```wgsl
struct VOut { @builtin(position) pos: vec4<f32> };

@group(0) @binding(0) var<uniform> color: vec4<f32>;

@vertex
fn vs_main(@builtin(vertex_index) i: u32) -> VOut {
    var p = array<vec2<f32>, 3>(
        vec2<f32>(-0.6, -0.5),
        vec2<f32>( 0.6, -0.5),
        vec2<f32>( 0.0,  0.7),
    );
    var out: VOut;
    out.pos = vec4<f32>(p[i], 0.0, 1.0);
    return out;
}

@fragment
fn fs_main() -> @location(0) vec4<f32> {
    return color;
}
```

## Step 2 — Compose from the catalog

The same `Shader::new` constructor takes a single URL, a single source string, *or* an array mixing any number of those — including **registry slugs**, short paths into FragmentColor's [shader catalog](/shaders/). Each slug resolves to a small pure WGSL function you can pull into your shader by name.

The composer fetches each slug (the registry is embedded by default), validates the merged source with naga, and links it all into one program. Slugs are content-addressed, so importing `noise/simplex2` from two different shaders compiles it once.

## Step 3 — Bring your own vertices

The triangle still lives inside the shader. Let's hand control back to the program. Two new objects from the core API:

- **`Vertex`** — one point in space. Carries a position and any extra attributes you want (color, UVs, normals).
- **`Mesh`** — a bag of vertices.

We attach the mesh to the shader, and the vertex stage reads from it instead of inventing coordinates.

<Snippet file="01-hello-triangle/03-vertex-gradient/example.rs" region="setup" />

Three vertices, each with its own position and color. The shader reads `position` at `@location(0)` and `color` at `@location(1)`, interpolates the color across the triangle's surface, and writes it out per pixel.

<Snippet file="01-hello-triangle/03-vertex-gradient/example.rs" region="shader" />

<Demo file="01-hello-triangle/03-vertex-gradient/example.rs" />

A smooth gradient — pink in one corner, green in another, blue at the top. Each vertex pulls its own color toward itself; the GPU blends the rest. You didn't write any "blend the colors" code — that's just what happens when the rasterizer interpolates a vertex output.

The interesting part isn't the gradient. It's that the same `Shader` instance is now drawing geometry the program decides on, and the change took five lines of Rust.

## Step 4 — A field of instances

One triangle is fine. Let's draw fifteen hundred.

We don't need fifteen hundred meshes. We need fifteen hundred *instances* of the same mesh — same three vertices, drawn over and over with a per-instance offset, phase, and tint. The GPU handles them all in a single draw call.

The new object is **`Instance`**. It carries attributes the same way a vertex does, but they apply to the whole copy of the mesh, not to one vertex of it.

<Snippet file="01-hello-triangle/04-particle-trail/example.rs" region="setup" />

The vertex shader takes both vertex and instance attributes. Vertex attributes drive the three corners of *one* triangle; instance attributes drive where the triangle goes and how it pulses. We also pass a `time` uniform so the GPU can wobble each instance independently.

The pulse rides through `easing/in_out_sine` — another catalog slug, pulled in next to the host source:

## Step 5 — Cinematic postfx

Let's make it look properly filmic. Render the particle scene into a *texture* instead of straight to the canvas, then hand that texture to a second shader that stacks four registry helpers on top:

- `postfx/chromatic_offsets` — three UVs for a subtle RGB channel split
- `color/tonemap_filmic` — Hejl/Burgess-Dawson highlight roll-off
- `postfx/vignette` — multiplier that darkens the edges
- `postfx/film_grain` — additive per-frame noise

Two more objects from the core API show up here:

- **`Pass`** — the unit of "render this to that". Lets you chain effects.
- **`Texture`** — the bridge between two passes (or a sampleable input).

## What just happened

In one tutorial we touched every object in FragmentColor's core API.

- **`Shader`** — the GPU program. Loaded from a URL in step 1, *composed from the catalog* in steps 2, 4, and 5.
- **`Vertex`** — a point with attributes. Step 3 introduced it.
- **`Mesh`** — a bag of vertices, optionally with instances. Step 3 used one vertex per corner; step 4 reused the same mesh with 1500 instances.
- **`Pass`** — the unit of rendering. Step 5 chained two of them: one drawing into a texture, one reading from it.
- **`Texture`** — sampleable image data on the GPU. Step 5 used a texture as the bridge between passes; the same object can also hold a loaded image, a CPU-uploaded byte buffer, or compute output.
- **`Renderer`** — the thing that actually draws. The same `render(...)` call worked for all five steps; everything else above just changes what goes through it.

That's the entire core. Compute shaders are a `Shader` with a `@compute` entry point — same object, different stage — and they slot into a `Pass` exactly like a graphics shader does.
