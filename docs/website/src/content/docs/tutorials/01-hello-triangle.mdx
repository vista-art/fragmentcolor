There's a rite of passage in graphics programming: **drawing your first triangle!**

**This typically takes thousands of lines of code.** Pick any tutorial for OpenGL, Vulkan, or raw WebGPU, and you'll spend an afternoon setting pixel formats, allocating command pools, and wiring descriptor sets & binding groups before any colored pixel reaches the screen.

**We're going to do it in six lines** of <Lang/>.

## Step 1 — Hello, triangle!

To draw anything anywhere, we need two things: a **surface** to draw to, and **something that draws**. In FragmentColor those are a `Target` (wraps a canvas or window) and a `Renderer`. Let's create them!

<MultiCode
  rust={`// canvas is a winit Window
let renderer = Renderer::new();
let target = renderer.create_target(canvas).await?;`}
  js={`// canvas is the <canvas> DOM element
const renderer = new Renderer();
const target = await renderer.createTarget(canvas);`}
  py={`# canvas is platform-specific (typically glfw)
renderer = Renderer()
target = await renderer.create_target(canvas)`}
  swift={`// canvas is platform-specific (UIView, NSView, ...)
let renderer = Renderer()
let target = try await renderer.createTarget(canvas)`}
  kotlin={`// canvas is platform-specific (SurfaceView, ...)
val renderer = Renderer()
val target = renderer.createTarget(canvas)`}
/>

Surface and Renderer are ready. Now we need to decide: *what* to draw?

That's what a **`Shader`** does. Shaders are tiny programs that run once per pixel. It's the smallest unit FragmentColor can render, and we don't need anything else to see cool things on screen. Let's grab a quick example from our  [shader catalog](/shaders/)  :

<ColorTriangle />

Congratulations! You just drew your first triangle!

Well, maybe it's not your first — I'm sure you remember that moment fondly, though.

What you'd actually see if you ran those three lines as a one-shot program is a single, static, pink triangle just sitting there — `set("color", ...)` once, `render(...)` once, then the program exits and the GPU goes home. Don't be fooled, though: those three lines are doing a lot. Behind the scenes, FragmentColor validated the WGSL, compiled a shader program, built a render pipeline, allocated GPU buffers to move data between CPU and GPU, created bind groups, set up a render pass, and dispatched the draw call. Our little shader runs **once for every pixel**. On a 1080p surface that's about 2 million invocations, all in parallel.

To turn the static frame into an animation, we run those last two lines (`set` and `render`) inside a render loop, changing the color input each tick:

Same shader handle, one new `set("color", ...)` per frame, one `render(...)` per frame. Every pixel gets the new color next frame. Press play:

<Demo file="01-hello-triangle/01-three-lines/example.rs" />

At 60 fps that's about 125 million pixel invocations per second, all driving the same triangle from a single uniform. We have one input. Time to give the program more to play with!

Before we move on, let's peek at the shader living at that URL — it's the other half of the story:

```wgsl
struct VOut { @builtin(position) pos: vec4<f32> };

@group(0) @binding(0) var<uniform> color: vec4<f32>;

@vertex
fn vs_main(@builtin(vertex_index) i: u32) -> VOut {
    var p = array<vec2<f32>, 3>(
        vec2<f32>(-0.6, -0.5),
        vec2<f32>( 0.6, -0.5),
        vec2<f32>( 0.0,  0.7),
    );
    var out: VOut;
    out.pos = vec4<f32>(p[i], 0.0, 1.0);
    return out;
}

@fragment
fn fs_main() -> @location(0) vec4<f32> {
    return color;
}
```

The `@group(0) @binding(0) var<uniform> color: vec4<f32>` line is what made `shader.set("color", ...)` work — FragmentColor reflected the binding right off the WGSL source. The `@vertex` stage runs three times and picks one of three positions. The `@fragment` stage returns the uniform as the pixel color. One triangle, one color.

## Step 2 — Compose from the catalog

Same `Shader::new` constructor, only this time we pass an *array*. It can mix any number of inline source strings, URLs, file paths, **or registry slugs** — short paths into FragmentColor's [shader catalog](/shaders/), each one a small pure WGSL function we can pull in by name.

The inline source calls `simplex2(...)` straight from the fragment stage as if it were a normal WGSL function — because, well, it now is:

`Shader::new` fetches each slug (the registry is embedded by default — no network), validates the merged source through naga, and links everything into one program. Slugs are content-addressed, so importing the same helper from two different shaders compiles it once and reuses it.

There are 225 of these helpers in the catalog right now — easing curves, color spaces, post-processing kernels, lighting models, hashes, SDFs in 2D and 3D. We'll keep dipping in for the rest of this tutorial.

## Step 3 — Bring your own vertices

Same triangle on screen, but now the program owns the geometry. Two new objects from the core API:

- **`Vertex`** — one point in space. Carries a position and any extra attributes you want to attach (color, UVs, normals, whatever).
- **`Mesh`** — a bag of vertices.

We hand the mesh to the shader, and the shader's vertex stage reads from it instead of making things up.

<Snippet file="01-hello-triangle/03-vertex-gradient/example.rs" region="setup" />

Three vertices, each with its own position and its own color. The shader reads `position` at `@location(0)` and `color` at `@location(1)`, interpolates the color across the surface of the triangle, and writes it out per pixel.

<Snippet file="01-hello-triangle/03-vertex-gradient/example.rs" region="shader" />

<Demo file="01-hello-triangle/03-vertex-gradient/example.rs" />

A smooth gradient — pink in one corner, green in another, blue at the top. Each vertex pulled the color toward itself and the GPU blended the rest. You didn't write any "blend the colors" code — that's just what happens when the rasterizer interpolates a vertex output.

The interesting thing here isn't the gradient. It's that the same `Shader` instance you had in step 1 is now drawing geometry the program decides on, and the change took five lines of <Lang />.

## Step 4 — A field of instances

One triangle is fine. Let's draw fifteen hundred.

We don't need fifteen hundred meshes. We need fifteen hundred *instances* of the same mesh — same three vertices, drawn over and over with a per-instance offset, phase, and tint. The GPU handles them all in a single draw call.

This is where the **`Instance`** object comes in. An instance carries attributes the same way a vertex does, but they apply to the whole copy of the mesh rather than to one vertex of it.

The shader pulls in `easing/in_out_sine` from the catalog so the per-instance pulse rides a smoother bell curve than a raw sine, and calls the imported function on the very last lines before assembling the output:

<Snippet file="01-hello-triangle/04-particle-trail/example.rs" region="shader" />

The vertex shader takes both vertex attributes and instance attributes. Vertex stuff drives the three points of *one* triangle; instance stuff drives where the triangle goes and what color it shifts towards. We also pass a `time` uniform so the GPU can wobble each instance independently using its phase.

Per frame, all we change is one number — the `time` uniform — and the shader animates every triangle from that:

## Step 5 — Run it through a cinematic postfx pass

Let's make it look properly cinematic! We're going to render the particle scene into a *texture* instead of straight to the canvas, then hand that texture to a second shader that stacks four registry helpers on top:

- `postfx/chromatic_offsets` — three UVs for a subtle RGB channel split
- `color/tonemap_filmic` — Hejl/Burgess-Dawson highlight roll-off
- `postfx/vignette` — multiplier that darkens the edges
- `postfx/film_grain` — additive per-frame noise

Two more objects from the core API show up here:

- **`Pass`** — the unit of "render this to that". Lets you chain effects.
- **`Texture`** — the bridge between two passes (or a sampleable input).

## What just happened

In one tutorial we touched every object in FragmentColor's core API. Each one earned its keep on a different step:

- **`Shader`** — the GPU program. Loaded from a URL in step 1, *composed from the catalog* in steps 2, 4, and 5; written inline alongside in step 3.
- **`Vertex`** — one point with attributes (position, color, anything you want). Step 3 introduced it.
- **`Mesh`** — a bag of vertices, optionally with instances. Step 3 used one vertex per corner; step 4 reused the same mesh with 1500 instances.
- **`Pass`** — the unit of rendering. Step 5 chained two of them: one drawing into a texture, one reading from it.
- **`Texture`** — sampleable image data on the GPU. Step 5 used a texture as the bridge between the two passes; the same object can also hold a loaded image, a CPU-uploaded byte buffer, or compute output.
- **`Renderer`** — the thing that actually draws. Same `render(...)` call worked for all five steps; everything else above just changes what gets rendered.

That's the entire core. Compute shaders are a `Shader` with a `@compute` entry point — same object, different stage — and they slot into a `Pass` exactly like a graphics shader does.
