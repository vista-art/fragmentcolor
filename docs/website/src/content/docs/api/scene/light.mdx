## Description

`Light` is the unified type covering the three glTF `KHR_lights_punctual`
kinds: **directional** (parallel rays â sun / sky / fill), **point**
(radiates from a position with inverse-square falloff â lamps / candles /
fireballs), and **spot** (point light constrained to a cone â flashlights /
headlamps / theatre lighting). The kind is set once at construction via
[`Light::directional`](/api/scene/light/directional),
[`Light::point`](/api/scene/light/point), or
[`Light::spot`](/api/scene/light/spot).

The unified surface keeps the public API simple: one type, one
[`Scene::add`](/api/scene/scene/add) method, one
[`Pass::add`](/api/core/pass/add) method, and one
binding per platform. Setters fall into two groups:

- **Universal** â [`set_color`](/api/scene/light/set_color)
  and [`set_intensity`](/api/scene/light/set_intensity)
  apply to every kind and return `Self` for chaining.
- **Kind-specific** â [`set_position`](/api/scene/light/set_position),
  [`set_direction`](/api/scene/light/set_direction),
  [`set_range`](/api/scene/light/set_range), and
  [`set_cone_angles`](/api/scene/light/set_cone_angles)
  return `Result<Self, LightError>`. Calling one on the wrong kind returns
  `LightError::FieldNotApplicable { kind, field }`.

The matching getters use `Option<T>` for the kind-specific fields:
[`position`](/api/scene/light/position),
[`direction`](/api/scene/light/direction),
[`range`](/api/scene/light/range),
[`inner_cone_angle`](/api/scene/light/inner_cone_angle),
[`outer_cone_angle`](/api/scene/light/outer_cone_angle)
return `Some` only on the kinds that carry the value, `None` otherwise.

Lights hold Arc-shared state, so a single handle can be absorbed by
multiple Passes; later mutators propagate to every shader the Light has
been wired into.

## Cost model

The default Material::pbr shader uses **forward shading** â every fragment
loops over every active light and accumulates the contribution. That makes
the per-frame cost roughly `O(fragments Ã lights)`, so the practical
ceiling depends on resolution and overdraw. The current cap of **32
lights** fits the forward path comfortably; for scenes with hundreds of
lights, a clustered / storage-buffer path is on the roadmap.

### Cap semantics â lights bind to the shader, not the Pass

The 32-light cap is enforced **per `ShaderObject`**, not per `Pass`. Two
Passes that share the same Material (and therefore the same shader)
share its light slots:

- Adding the same `Light` to Pass A and Pass B reuses one shader slot
  (the dedup is by shader-pointer identity). Both Passes render with
  that light visible.
- Adding two different `Light`s â one to Pass A, one to Pass B â packs
  them into two separate slots on the shared shader. **Both lights
  illuminate both Passes**, even though each was only `add(&...)`-ed
  to one. That's surprising if you think of a Pass as an isolated
  scene; it's the right answer if you think of a Material as the
  lighting model and the Passes as views into the same lit world.
- The cap exceeds at 32 *distinct* Lights on one shader. The 33rd
  returns `Err(PassError::LightCapReached { cap: 32 })`.

For genuinely independent lighting setups, build a separate Material
per scene (each `Material::pbr()?` allocates a fresh shader, so its 32
slots are independent) or pass your own shader to `Material::custom`.
A single scene with one Material covers the typical case â the per-
shader semantic is what lets `Scene::add(&light)` propagate through
every Pass that renders the Scene without re-attaching.

| `spot`        | point light constrained to a cone                           |

### Light::directional

Construct a directional light. `direction` is the world-space vector the
light travels along (so `[0, -1, 0]` is straight down â noon sun); `color`
is linear-RGB intensity (`[1, 1, 1]` for full white, scaled values for
dimmer or tinted lights).

Use this for sun / moon / fill / key lights â anything where every shaded
surface should receive parallel rays from a fixed direction. For positioned
sources, see [`Light::point`](/api/scene/light/point)
and [`Light::spot`](/api/scene/light/spot).

A directional light carries no position. Calling
[`set_position`](/api/scene/light/set_position),
[`set_range`](/api/scene/light/set_range), or
[`set_cone_angles`](/api/scene/light/set_cone_angles)
on a directional light returns `LightError::FieldNotApplicable`.

#### Example

assert_eq!(sun.color(), [1.0, 0.95, 0.9]);

### Light::point

Construct a point light. `position` is the world-space location the light
radiates from; `color` is linear-RGB intensity (`[1, 1, 1]` for full white).
Falloff is inverse-square distance â bring
[`set_range`](/api/scene/light/set_range) in to
cap the influence radius, and
[`set_intensity`](/api/scene/light/set_intensity)
to scale the color uniformly.

Use this for lamps, candles, fireballs â anything where light radiates
outward from a fixed point. For a parallel-ray sun-like light see
[`Light::directional`](/api/scene/light/directional);
for a cone-constrained beam see
[`Light::spot`](/api/scene/light/spot).

A point light carries no direction or cone. Calling
[`set_direction`](/api/scene/light/set_direction)
or [`set_cone_angles`](/api/scene/light/set_cone_angles)
on a point light returns `LightError::FieldNotApplicable`.

### Light::spot

Construct a spot light. `position` is the world-space location the light
radiates from; `direction` is the world-space vector the cone aims along
(the cone axis is `-direction`); `color` is linear-RGB intensity.

Falloff is inverse-square distance with a smooth cone roll-off between an
inner and outer cone angle (defaults: `0` and `Ï/4`). Tighten the beam
with [`set_cone_angles`](/api/scene/light/set_cone_angles),
cap the influence with
[`set_range`](/api/scene/light/set_range), and
scale the radiance with
[`set_intensity`](/api/scene/light/set_intensity).

Use this for flashlights, headlamps, theatre lighting, mood key-lights â
anything where the lit volume is a cone aimed at a target. For an
unconstrained point source see
[`Light::point`](/api/scene/light/point);
for parallel-ray illumination see
[`Light::directional`](/api/scene/light/directional).

A spot light carries every kind-specific field, so every kind-specific
setter (`set_position`, `set_direction`, `set_range`, `set_cone_angles`)
succeeds with `Ok(self)`.

#### Example

<Tabs>

<TabItem label="Rust">
<Code
code={`
fn main() -> Result<(), Box<dyn std::error::Error>> {
use fragmentcolor::Light;

let torch = Light::spot([0.0, 1.8, 1.0], [0.0, -0.3, -1.0], [1.0, 0.9, 0.7])
    .set_intensity(5.0)
    .set_cone_angles(0.15, 0.4)?;

assert_eq!(torch.position(), Some([0.0, 1.8, 1.0]));
assert_eq!(torch.direction(), Some([0.0, -0.3, -1.0]));
assert!((torch.outer_cone_angle().unwrap() - 0.4).abs() < 1.0e-6);
Ok(())
}
`}

### Light::kind

Read which kind this Light was constructed as. Returns a `LightKind` value
(`Directional`, `Point`, or `Spot`). Bindings expose this as a string
literal (`"directional"`, `"point"`, `"spot"`) so the kind tag round-trips
the same way it does in glTF's `KHR_lights_punctual`.

Use this when inspecting a Light built elsewhere (e.g. loaded from a
glTF scene) to decide which kind-specific setters or getters to call.

#### Example

<Tabs>

<TabItem label="Rust">
<Code
code={`

use fragmentcolor::{Light, LightKind};

let sun = Light::directional([0.0, -1.0, 0.0], [1.0, 1.0, 1.0]);
let bulb = Light::point([0.0, 2.5, 0.0], [1.0, 1.0, 1.0]);
let torch = Light::spot([0.0, 1.8, 1.0], [0.0, -1.0, 0.0], [1.0, 1.0, 1.0]);
assert_eq!(sun.kind(), LightKind::Directional);
assert_eq!(bulb.kind(), LightKind::Point);
assert_eq!(torch.kind(), LightKind::Spot);
Ok(())
}
`}

### Light::position

light (directional rays travel from infinitely far away â no position to
report). Call
[`set_position`](/api/scene/light/set_position)
to update it.

#### Example

<Tabs>

<TabItem label="Rust">
<Code
code={`
